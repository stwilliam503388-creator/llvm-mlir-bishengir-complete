## 2.1 计算机怎么"理解"你的代码？

面对这行字符串：

```
a + b * 3
```

计算机看到的是 `['a', ' ', '+', ' ', 'b', ' ', '*', ' ', '3']`——一串字符，毫无意义。
编译器要做第一件事：**从字符串中提取出结构**。

它需要知道：
- `a` 和 `b` 是变量名
- `3` 是一个数字
- `+` 和 `*` 是运算符
- **先乘后加**（运算符优先级）

这个提取结构的过程叫**解析（Parsing）**。
解析的结果是一棵**树**。

---

## 2.2 AST — 代码的语法树

AST（Abstract Syntax Tree，抽象语法树）——代码结构的树形表示。

```
表达式: a + b * 3

解析成 AST:

         ┌──── [+]
         │      │
       ┌─┴─┐    │
       a   │   ┌─┴─┐
           │   b   3
           │
     BinOp(+)         ← 根节点: 加法运算
     ├── Ident(a)     ← 左子节点: 变量 a
     └── BinOp(*)     ← 右子节点: 乘法运算
          ├── Ident(b) ← 子节点: 变量 b
          └── Num(3)   ← 子节点: 数字 3
```

为什么要建成树？

| 好处 | 说明 |
|------|------|
| **优先级自然体现** | `*` 在 `+` 的子树里，所以先算 |
| **括号自动消解** | `(a+b)*3` 会生成不同的树（`+` 在下面） |
| **便于遍历** | 从根节点往下走，能访问所有子表达式 |
| **便于修改** | 想替换 `b*3` 为 `c`，只需换掉一个子树 |

### 在项目中找到它

打开 `projects/toy-mini/toymini.cpp`，搜索 `struct NumberExpr`：

```cpp
// AST 节点：数字
struct NumberExpr : ASTNode {
    double val;
    std::string toString(int indent) const override {
        return std::string(indent, ' ') + "Num(" + std::to_string(val) + ")";
    }
};

// AST 节点：二元运算
struct BinaryExpr : ASTNode {
    std::string op;                      // "+", "-", "*"
    std::unique_ptr<ASTNode> lhs, rhs;    // 左子树、右子树
};
```

这就是 `a + b * 3` 的 C++ 表达——`BinaryExpr("+", Ident("a"), BinaryExpr("*", Ident("b"), Num(3)))`。

**AST 的特点**：保留了所有语法信息（变量名、运算符、括号结构），但**不含执行信息**。

## 2.3 IR — 中间表示

AST 很方便人类理解，但不方便机器优化和执行。因为：
- 树结构**遍历慢**（要递归访问每个节点）
- 树的**依赖关系隐含**在层级中（没有显式的数据流）

所以编译器会把 AST **拍平**成线性的指令序列——这就是 IR。

```
AST:                                IR (三地址码):
                                      │
         ┌──── [+]                    ├── t1 = b * 3
         │      │                     ├── t2 = a + t1
       ┌─┴─┐    │                     └── result = t2
       a   │   ┌─┴─┐
           │   b   3

             ↓ 拍平

     t1 = mul(b, 3)     ← 临时变量 t1 存 b*3 的结果
     t2 = add(a, t1)    ← 临时变量 t2 存 a+t1 的结果
     result = t2
```

这种形式叫**三地址码（Three-Address Code，TAC）**——每个指令最多有一个运算符，操作数最多三个。

在此基础上，现代编译器采用 **SSA（Static Single Assignment，静态单赋值）** 形式：

**规则：每个变量只赋值一次。**

```llvm
; SSA 形式的 LLVM IR
%1 = mul i32 %b, 3        ; 结果叫 %1，以后不会再有另一个 %1
%2 = add i32 %a, %1       ; 结果叫 %2，以后不会再有另一个 %2
```

这看起来像废话，其实解决了大问题：

| 场景 | 非 SSA | SSA |
|------|--------|-----|
| 一个变量被改了多次 | `x = 1; x = 2;` → 不知道最后是哪个 | `x1 = 1; x2 = 2;` → 明确知道 |
| 判断数据依赖 | 需要做"到达定值分析" | **一眼就能看**：`%2` 依赖 `%1` 和 `%a` |
| 优化 | 需要复杂分析才能确认安全 | **安全保证**：改 `%2` 不会影响 `%1` |

**SSA 让数据流变得可见。** 这是 LLVM IR 的核心设计。

### 在项目中找到它

打开 `projects/standalone-mlir/test/example.mlir`：

```mlir
// MLIR 也是 SSA 形式
%0 = standalone.constant { value = 3.0 : f64 } : tensor<2x3xf64>
%1 = standalone.transpose(%0) : tensor<2x3xf64> -> tensor<3x2xf64>
%2 = standalone.add %0, %1 : tensor<2x3xf64>, tensor<3x2xf64> -> tensor<3x2xf64>
```

每个 `%N` 都是只赋值一次的 SSA 变量。

## 2.4 AST vs IR 对比

| 维度 | AST（抽象语法树） | IR（中间表示） |
|------|------------------|---------------|
| **形状** | 树形 | 线性（指令序列） |
| **保留什么** | 语法结构（括号、优先级） | 数据流（依赖关系） |
| **去掉什么** | - | 语法糖（变量名 `a` 变成 `%a`） |
| **适合** | 语法分析（告诉我写了什么） | 优化和执行（告诉我怎么做） |
| **类比** | 建筑设计图（标注了每面墙的位置） | 施工指令单（先砌这面墙再装那扇窗） |
| **在本项目中** | `toymini.cpp` 的 AST.h | `example.mlir`、`.ll` 文件 |

## 2.5 快速自测

1. **AST 是树还是列表？为什么？**
   - 答：树。因为代码有嵌套结构（`a + b * 3` 中的 `*` 在 `+` 里面），树能自然表达嵌套

2. **SSA 的"每个变量只赋值一次"有什么好处？**
   - 答：一眼就能看出数据依赖关系——`%2` 用了 `%1`，那么 `%1` 一定在 `%2` 之前定义

3. **在 toy-mini 项目中，`a + b * 3` 被解析成什么？**
   - 答：`BinaryExpr("+", Ident("a"), BinaryExpr("*", Ident("b"), Num(3)))`
   - 这就是 AST 节点

4. **IR 之于 AST，就像 _____ 之于 _____？**
   - 参考：菜谱的"食材清单"之于"烹饪步骤"
   - AST = 食材清单（有什么），IR = 烹饪步骤（怎么做）
