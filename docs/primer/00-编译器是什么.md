## 1.1 一条代码的执行之路

先看三段代码：

```python
# Python
a = 3
b = 5
c = a + b
```

```cpp
// C++
int a = 3;
int b = 5;
int c = a + b;
```

```python
# Triton（基于 Python 语法，但会被编译成 GPU/NPU 指令）
@triton.jit
def add_kernel(A, B, C, N):
    pid = tl.program_id(0)
    offset = pid * 128
    a = tl.load(A + offset, mask=offset < N)
    b = tl.load(B + offset, mask=offset < N)
    c = a + b
    tl.store(C + offset, c, mask=offset < N)
```

这三种语言都在做同一件事——**让计算机算出加法**。但它们的**执行方式完全不同**：

| 语言 | 执行方式 | 类比 |
|------|---------|------|
| **Python** | **解释执行**：读一行 → 执行一行 → 再读下一行 | 同声传译：一边听一边翻译 |
| **C++** | **编译执行**：全部读完 → 全部翻译成机器码 → 再执行 | 笔译：先通读全文再翻译 |
| **Triton** | **JIT 编译**：运行时即时编译成 GPU/NPU 指令 | 现场口译：听到后快速翻译，但翻译结果会缓存复用 |

关键问题：**为什么 Triton 需要显式"编译"而 Python 看起来不用？**

> 注意：Python 其实也有编译步骤——它会先编译成字节码（`.pyc`），再由解释器逐条执行字节码。但这个过程对用户透明，且目标是通用的虚拟机字节码，不是特定硬件指令。

核心区别在于目标硬件：
- Python → **CPU**（x86/Arm 指令集，通过通用虚拟机间接执行）
- Triton → **GPU/NPU**（NVIDIA/华为，每种芯片指令完全不同，必须编译成专用指令）

你不能用写 Python 的方式直接控制 NPU——人和芯片说的是**不同的语言**。
编译器就是那个**翻译官**。

## 1.2 编译器是一个翻译官团队

```
你写的代码 (Python/Triton)        ← 你的语言
       │
       ▼
┌──────────────────────┐
│   前端 Frontend       │  ← 读懂你的代码
│   (负责: 语法分析)    │     英语老师分析句子结构
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│   中端 Middle-end     │  ← 优化代码
│   (负责: 变换和优化)  │     把"3+5+2"换成"10"
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│   后端 Backend        │  ← 生成目标代码
│   (负责: 指令生成)    │     中文→英文翻译
└──────────┬───────────┘
           ▼
机器码 (NPU 指令)             ← 芯片的语言
```

这个三阶段架构——前端 / 中端 / 后端——是所有现代编译器的基本骨架。

**为什么非要分三步？**

假设你要把一篇中文文章翻译成英文、日文、法文三个版本：
- **笨办法**：找三个人，每人从中文直接翻（3 个前端 × 3 个后端 = 9 个人的工作）
- **聪明办法**：先把中文翻译成中间笔记（比如 Esperanto），再从 Esperanto 翻成英/日/法（1 个前端 + 1 个中间表示 + 3 个后端 = 5 个人的工作）

编译器分三步，**核心目的就是为了复用**。

LLVM 的厉害之处：它定义了**一种通用的中间表示（IR）**。
只要你能把你的语言翻译成 LLVM IR，LLVM 就能自动帮你生成 x86、Arm、RISC-V 等几十种 CPU 的机器码。

```
  你的代码                   LLVM IR                    机器码
  Python/C++/Rust  ──→  %1 = add i32 %a, %b  ──→  x86: add eax, ebx
                        (通用中间形式)               Arm: ADD R0, R1, R2
                                                     RISC-V: ADD a0, a1, a2
```

**本项目的 MLIR** 则是 LLVM 的升级版——它不只有一种中间表示，而是有**多种**。
因为 AI 芯片的多样性太强了，一种 IR 不够用。

## 1.3 编译器不只是翻译——它还要优化

"优化"这个词在编译器里很微妙。**它不是你写代码时想的"优化性能"**。

编译器的优化规则是：
**你的代码功能不变，我换一种方式实现，让它跑得更快。**

```python
# 你写的代码：
c = a + b + c + d
#       ↑
# 如果 a=3, b=5，编译器可能变成：
t = 8        # 常量折叠：加法直接算好
c = t + c + d
```

常见优化技术（你会在本项目的 Pass 中看到）：

| 优化 | 做什么 | 类比 |
|------|--------|------|
| 常量折叠 (Constant Folding) | `3+5` 直接算成 `8` | 菜谱里 "3个鸡蛋+5个蛋" 直接写 "8个蛋" |
| 死代码消除 (Dead Code Elimination) | 删掉从不执行的分支 | 菜谱里 "如果今天是闰年就加盐" → 删除 |
| 循环展开 (Loop Unrolling) | 小循环直接重复写 | "重复3次加盐" → "加盐、加盐、加盐" |
| 指令调度 (Instruction Scheduling) | 重排指令顺序 | 先切菜再烧水，不等水干烧 |
| 向量化 (Vectorization) | 一次处理多个数据 | 不是逐根切葱，而是整把切 |

优化为什么重要？对于一个 AI 模型推理，编译器优化可能带来 **2-5 倍** 的性能差距。

## 1.5 快速自测

读完本节后，你应该能回答：

1. **编译器和解释器的本质区别是什么？**
   - 答案：解释器读一行执行一行（Python），编译器全部读完再执行（C++），JIT 编译介于两者之间（Triton）

2. **为什么编译器要分前端/中端/后端？**
   - 答案：为了复用。中端定义一个通用 IR，支持新语言只需新前端，支持新硬件只需新后端

3. **LLVM IR 和 MLIR 是什么关系？**
   - 答案：LLVM IR 是一种通用 IR，MLIR 是支持多种 IR（dialect）的框架。MLIR = 多个 LLVM IR 形态的组合

4. **编译器的"优化"是优化你的代码逻辑吗？**
   - 答案：不是。编译器不改变你的算法，只是换一种实现方式让它跑得更快
