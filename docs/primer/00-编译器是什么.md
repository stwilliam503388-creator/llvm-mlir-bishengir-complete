## 一个翻车现场

你写了一段 Triton kernel：

```python
@triton.jit
def my_kernel(x, y):
    pid = tl.program_id(0)
    offset = pid * 32 + tl.arange(0, 32)
    # ... 看起来很对的代码
```

跑了，挂了。报错信息：

```
TTIR → TritonGPU → LLVM → PTX: error at lowering pass
```

你完全看不懂。**因为 Triton 不是一个"直接运行"的语言——它背后有一套完整的编译器流水线。** 不理解编译器，你就看不懂报错，不会调优，不知道哪里出了问题。

这就是你学编译器的原因——不是要自己写编译器，而是**能看懂编译器在干什么**。

## 1.1 三行代码，三种执行方式

```python
a = 3; b = 5; c = a + b    # Python
```

```cpp
int a = 3, b = 5, c = a + b;  // C++
```

```python
@triton.jit
def add_kernel(a, b, c):      # Triton
    c = a + b
```

三个语言都在做同一件事——算出 `c = 8`。但计算机"看懂"它们的方式完全不同：

| 语言 | 执行方式 | 通俗理解 |
|------|---------|---------|
| Python | **解释执行** | 一行一行翻译成机器指令，翻一行跑一行 |
| C++ | **编译执行** | 整个文件翻译成机器指令（.exe），再运行 |
| Triton | **编译 + GPU 执行** | 先翻译成 GPU 指令，再送到显卡上跑 |

**解释器**像同声传译——边听边翻。**编译器**像笔译——拿到全文，翻译好，再给你看。

> ✅ **检查自己**：你能一句话说出编译器和解释器的区别吗？
> → 解释器一行一行翻译执行；编译器整个文件翻译完再执行。

## 1.2 编译器为什么要拆成三步

```text
源代码（你写的）     →    中间表示（IR）     →    机器码（硬件执行的）
    ↑ 前端               ↑ 中端               ↑ 后端
```

看起来多了一步，但这是故意的：

| 如果只要两步 | 如果要三步 |
|------------|-----------|
| Python → x86 | Python → IR → x86 |
| Python → ARM | Python → IR → ARM |
| Python → NPU | Python → IR → NPU |
| **换硬件就要重写编译器** | **只换后端，前端和中端重用** |

**IR（中间表示）** 就是编译器的"通用语"。就像你写中文，翻译成英文笔记，再翻译成日文——英文笔记不是给谁看的，是中间产物。有了它，换目标语言只需换最后一步。

---

## 1.3 三步分别做什么

```text
你写的代码         IR（中间表示）         机器码
┌─────────┐      ┌──────────────┐      ┌─────────┐
│ a + b * 3│  →  │ %1 = mul b, 3│  →  │ mov eax │
│         │      │ %2 = add a, %1│     │ mul ecx │
└─────────┘      └──────────────┘      └─────────┘
   前端               中端                后端
  (解析代码)        (做优化)           (生成机器码)
```

- **前端**：把你写的代码变成 IR（"翻译成通用语"）
- **中端**：在 IR 上做优化（"改写法，不改意思"）
- **后端**：把 IR 变成目标硬件的机器码（"从通用语翻译成方言"）

> ✅ **检查自己**：
> 1. 为什么要有 IR 这一步而不是直接从源码到机器码？
>    → 换硬件只换后端，前端和中端可以重用。
> 2. Triton 的编译器做了哪三步？
>    → Triton Python → TTIR（IR）→ TritonGPU → LLVM → PTX（GPU 指令）。

## 1.4 优化是什么

优化是**不改逻辑，只改实现方式**。

```text
a = b + 0   →   优化后 →   a = b       （多此一举，去掉）
a = 2 * b   →   优化后 →   a = b + b   （乘法变加法，在某些 CPU 上更快）
```

编译器在 IR 阶段做这些优化，因为它们不依赖你写的具体语言，也不依赖最终跑在什么硬件上——是"中间表示"这个抽象层让优化变得通用。
